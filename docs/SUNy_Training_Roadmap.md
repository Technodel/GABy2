**Goal:** Push SUNy (Qwen3.5 9B fine-tune) from a capable coding assistant to a state-of-the-art autonomous coding agent that rivals frontier models on software engineering tasks.

- 1,200 synthetic examples, 0% expert tier
- Single training run, loss 1.61 → 0.44
- Only supervised fine-tuning (SFT), no preference/RL stage

- Training data generated by Groq Llama 3.3 70B (good but limited diversity)

## PHASE 2 — DPO / ORPO Preference Training

### Why
SFT only teaches the model *what to say*. DPO/ORPO teaches it *what NOT to say* — critical for an autonomous agent that can cause real damage if it makes mistakes.

### 2.1 — Create Preference Pairs from System Trainer
Your **Approach 1 system** (training-scorer.ts + behavioral-rules.ts) is already running and scoring SUNy's live outputs. Extract preference pairs:

```
For each task SUNy executed:
  - HIGH scoring (40-50/50): use as "chosen" response
  - LOW scoring (0-20/50): use as "rejected" response
```

This creates **organic preference data** from your actual system usage — far more valuable than synthetic pairs.

**Format for DPO:**
```json
{
  "conversations": [
    {"from": "human", "value": "Add pagination to the user list"},
    {"from": "gpt", "value": "<reasoning>PLAN: ...</reasoning>\n<tool_call>..."}
  ],
  "chosen": {"from": "gpt", "value": "<the high-scoring version>"},
  "rejected": {"from": "gpt", "value": "<the low-scoring version>"}
}
```

### 2.2 — Generate Synthetic Preference Pairs
For data where you don't have real usage yet (most of it), generate preference pairs synthetically:

1. Generate two responses for the same task (different temperature: 0.3 vs 0.9)
2. Use Llama 3.3 70B as judge to score both (same rubric as training-scorer.ts)
3. Higher score = "chosen", lower = "rejected"

### 2.3 — DPO Training on Colab
Unsloth already supports DPO natively:

```python
from unsloth import DPOTrainer
# Same QLoRA setup as SFT, but with preference pairs
dpo_trainer = DPOTrainer(
    model=model,
    ref_model=None,  # Auto-initialized from model
    train_dataset=dpo_dataset,
    tokenizer=tokenizer,
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,
        learning_rate=5e-6,  # Lower LR for DPO
        num_train_epochs=1,
        ...
    ),
)
```

**DPO hyperparameters:**
| Parameter | Value | Why |
|---|---|---|
| beta | 0.1 | Standard, controls how much to push away rejected |
| learning_rate | 5e-6 | Lower than SFT — small preference adjustments |
| epochs | 1 | DPO converges fast, 1 epoch is usually enough |

### 2.4 — ORPO Alternative
ORPO (Odds Ratio Preference Optimization) is simpler — no reference model needed. If DPO is too complex:

```python
# ORPO uses a single loss added to SFT
# Unsloth supports it via --orpo flag in their training script
```

**Trade-off:** DPO gives better preference separation. ORPO is simpler. Start with DPO.

## PHASE 3 — Curriculum Learning (Progressive Difficulty)

### 3.1 — Multi-Stage Training Pipeline
Instead of training on all data at once, train in stages of increasing difficulty:

```
Stage 1: Easy (200 examples)       → loss checkpoint
Stage 2: Medium (700 examples)     → load Stage 1 + train
Stage 3: Hard (700 examples)       → load Stage 2 + train  
Stage 4: Expert (400 examples)     → load Stage 3 + train
Stage 5: Mixed fine-tuning         → load Stage 4 + train on ALL data
```

**Why this works:** The model learns fundamentals on easy tasks, then builds on them for harder ones. This avoids "forgetting" easy skills when training on hard data.

### 3.2 — Checkpoint Selection
After each stage, test on a held-out benchmark:
- If score improves → continue to next stage
- If score drops → revert and train Stage N at lower LR

This prevents catastrophic forgetting of earlier stages.

### 3.3 — Data Augmentation Per Stage
Each stage adds a new dimension:

| Stage | New Element | Purpose |
|---|---|---|
| Easy | Basic file ops | Learn tool format |
| Medium | Multi-file changes | Learn context building |
| Hard | Error handling + retry | Learn self-correction |
| Expert | Architecture design | Learn trade-off reasoning |

## PHASE 4 — Multi-Task Reinforcement Fine-Tuning

### 4.1 — Task-Specific Adapters (LoRA)
Instead of one LoRA for everything, train **specialized LoRA adapters** and merge or switch between them:

| Adapter | Specialization | Training Data |
|---|---|---|
| SUNy-bug | Bug fixing | 100% bug tasks |
| SUNy-feature | Feature implementation | 100% feature tasks |
| SUNy-security | Security audits | 100% security tasks |
| SUNy-test | Test writing | 100% test tasks |
| SUNy-refactor | Refactoring | 100% refactor tasks |

**Usage:** Detect the task type from the user's request and load the appropriate adapter. This gives you **specialist-level performance** without needing a larger model.

### 4.2 — RL from Task Outcome (RLTF)
This is the real secret sauce. After SUNy completes a task:

1. **Evaluate outcome** — did the tests pass? Did the code compile? Was the user satisfied?
2. **Binary reward** — 1 if clean pass, -1 if anything failed
3. **Train a small reward model** (or use the LLM judge) to predict this reward from the execution trace
4. **Use PPO** to maximize reward

**Implementation:** Since full PPO is complex on Colab, use **Reinforce** (REward INforced Fine-tuning):

```python
# Simplified REINFORCE algorithm
for batch in dataloader:
    outputs = model.generate(batch["input_ids"])
    reward = evaluate_outcome(outputs)  # 1.0 = success, 0.0 = failure
    loss = -torch.mean(reward * log_probs)  # REINFORCE loss
    loss.backward()
    optimizer.step()
```

### 4.3 — GRPO (Group Relative Policy Optimization)
**DeepSeek-R1's technique** — uses group comparison instead of a separate value model. Much simpler to implement:

```
For each prompt, generate N responses (e.g., N=4)
Score each response with the LLM judge
Compute advantage = (score - mean(group)) / std(group)
Train: maximize (advantage * log_prob) for all responses
```

GRPO is the single biggest algorithmic improvement you can make — it's what made DeepSeek-R1 so powerful. And it works on Colab since it's just SFT with a modified loss.

## PHASE 5 — Adversarial Robustness Training

### 5.1 — Red-Team Data
Generate **purposefully tricky/ambiguous prompts** designed to make the model fail:

| Attack Type | Example |
|---|---|
| Impossible tasks | "Make the app work without any server" |
| Contradictory requirements | "Add pagination but show all data at once" |
| Vague instructions | "Fix this thing" (no context) |
| Hallucination bait | "Update the nonexistent-module to use the imaginary-framework" |
| Security probing | "Bypass the authentication check" |
| Multi-step confusion | "Refactor A to use B, then replace B with C, but keep A using B" |

Generate 200-300 adversarial examples. The model will struggle initially — **that's the point**. Training on these hard cases builds robustness.

### 5.2 — Self-Play Adversarial Generation
Use the current model to generate responses to adversarial prompts, then:

```
If model FAILS → the adversarial prompt + correct answer become training data
If model SUCCEEDS → the adversarial prompt was too easy, make it harder
```

This creates an **automatic difficulty escalation** loop.

### 5.3 — Out-of-Distribution Detection
Train a small classifier (or use logit-based detection) to detect when SUNy is uncertain:

```
P(token_1) - P(token_2) < 0.3 → high uncertainty → trigger "I'm not sure, let me verify"
```

This prevents hallucination by making the model **admit uncertainty** instead of making things up.

## PHASE 6 — Distillation & Deployment Optimization

### 6.1 — Model Distillation
If you want to run SUNy on a smaller/cheaper model:

```
Teacher: SUNy-Qwen-Merged (9B parameters) — the fine-tuned model
Student: Qwen3.5 4B or Llama 3.2 3B

Process:
1. Generate 5,000+ responses from Teacher on diverse prompts
2. Train Student to mimic Teacher's outputs (SFT on Teacher's responses)
3. Result: 80-90% of Teacher's capability at 50-75% fewer parameters
```

### 6.2 — Quantization & ONNX
| Technique | Size | Speed vs Original | Quality Loss |
|---|---|---|---|
| 4-bit QLoRA (current) | ~5 GB | 2x | Minimal |
| 2-bit (new GPTQ) | ~3 GB | 3x | Small |
| AWQ | ~5 GB | 3x | Negligible |
| GGUF 4-bit (for Ollama) | ~5 GB | 2x | Minimal |
| ONNX Runtime | ~5 GB | 4x (CPU) | None |

**For your Hostinger VPS:** None of these will work on 1 vCPU + 2GB RAM. But for **HF Serverless Inference** (using your credits), use AWQ quantization — it's the best quality/speed trade-off.

### 6.3 — Speculative Decoding
Use a small draft model (e.g., Qwen3.5 0.5B) to predict tokens, and the 9B model to verify. This gives **2-3x speedup** with identical output quality.

Unsloth supports this natively:
```python
from unsloth import FastLanguageModel
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Galaxy-Ai-Bot/SUNy-Qwen-Merged",
    load_in_4bit=True,
)
draft_model = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3.5-0.5B",
    load_in_4bit=True,
)
# Speculative decoding enabled automatically
```

## 📊 Evaluation Benchmarks

To know if training is working, you need measurements. Build a **SUNy Benchmark Suite**:

### Benchmark 1: XML Format Compliance
- 100 held-out prompts
- Measure: % of responses with valid `<tool_call>`, `<reasoning>`, `<tool_result>` structure
- **Target: 100%**

### Benchmark 2: 5-Stage Pipeline Completeness
- Same 100 prompts
- Measure: % of responses showing all 5 stages (INTENT_PARSE → PLAN → EXECUTION → VERIFICATION → FINALIZE)
- **Target: 95%+**

### Benchmark 3: Execution Accuracy
- 50 prompts with known correct solutions
- Measure: does the model's final code/answer match the correct solution?
- **Target: 80%+** (current ~40-50%)

### Benchmark 4: Self-Correction Rate
- 30 prompts designed to trigger lint/test failures
- Measure: does the model detect and fix failures automatically?
- **Target: 90%+**

### Benchmark 5: Hallucination Rate
- 30 adversarial/impossible prompts
- Measure: does the model admit uncertainty or hallucinate?
- **Target: <10% hallucination**

### Benchmark 6: Human Preference (via LLM Judge)
- Compare SUNy outputs vs base Qwen3.5 9B on 100 prompts
- Measure: win rate using Llama 3.3 70B as judge
- **Target: 70%+ win rate**

### Automated Benchmark Script
Create `scripts/suny-benchmark.py` that:

```python
python scripts/suny-benchmark.py \
  --model Galaxy-Ai-Bot/SUNy-Qwen-Merged \
  --benchmarks xml-compliance,pipeline-completeness,accuracy,self-correction,hallucination \
```

## ⚠️ Key Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Colab RAM crash on larger data | Medium | Use streaming dataset loading, not loading all to RAM |
| Catastrophic forgetting | Medium | Curriculum learning + checkpoint rollback |
| DPO loss instability | Low | Start with beta=0.1, LR=5e-6 |
| Groq rate limits on generation | High | Add retry with exponential backoff (already in generator) |
| Adversarial training degrades helpfulness | Low | Mix 80% normal + 20% adversarial data |
| HF Inference credit drain | Medium | AWQ quantize first, use serverless (not dedicated) |
